1. What is this application?

A browser-based tool for managing your entire printer fleet from one place.

Printer Asset Management is a self-hosted web application built to give IT teams and administrators full visibility and control over their printer infrastructure. Instead of spreadsheets and scattered records, everything lives in one place: hardware details, consumable stock, maintenance history, supplier contracts, and budgets.

Key Features

ModuleWhat it does
DashboardLive overview of fleet health, stock levels, contract expiries, and budget usage.
PrintersFull inventory of all printers — model, location, status, service dates, page counts, and assigned consumables.
ConsumablesTrack toner, ink, drums, and other supplies. Monitor stock levels and assign items to specific printers.
ContractsStore and track supplier contracts with expiry alerts and PDF attachment support.
SuppliersManage supplier contact details and logos.
MaintenanceWork order system — create, track, and complete maintenance jobs with full status history.
CAPEX / OPEXCapital and operational expenditure tracking with depreciation schedules and cost-per-page analysis.
BudgetSet annual budgets and track spending in real time.
ReportsFull activity log of every change made in the system, exportable to CSV.
AdminUser management, email/SMTP configuration, and notification alerts.
💡
Self-hosted means full control

The application runs entirely on your own hardware (your Synology NAS). No data is sent to any third-party cloud. You own everything.

Why these technologies?

React + Vite (Frontend)

React is a popular JavaScript library for building user interfaces. Vite is the build tool that compiles the React code into static HTML/CSS/JS files that a browser can load. TypeScript adds type safety, catching errors before they become bugs. The frontend communicates with the backend via a REST API.

Laravel (Backend)

Laravel is a PHP web framework that handles all business logic, database operations, authentication, and API responses. It runs as a PHP-FPM process inside Docker — meaning it waits for requests and processes them efficiently. Laravel Sanctum handles authentication using Bearer tokens.

MySQL (Database)

MySQL stores all persistent data: printers, consumables, contracts, users, activity logs, etc. It runs in its own Docker container and is never exposed directly to the internet — only the backend container can talk to it.

Nginx (Web Server)

Nginx serves two purposes: it serves the built React frontend as static files, and it proxies all /api/ requests to the Laravel PHP-FPM backend. This single entry point on port 8900 handles both the web app and the API.

Docker

Docker packages each part of the application into an isolated container. This means the app runs identically on any machine regardless of what's installed on it. Docker Compose orchestrates all three containers (database, backend, nginx) together.

5. Preparing the NAS

One-time setup steps on the Synology DSM interface.

1

Install required packages

Open Package Center in DSM. Search for and install both:

  • Container Manager — the Docker interface
  • Git Server — provides the git command in SSH
2

Enable SSH

Go to Control Panel → Terminal & SNMP. Check Enable SSH service and click Apply.

SSH lets you type commands on the NAS remotely from your computer's terminal.

3

Connect via SSH

On your Mac or PC, open a terminal and type:

ssh your_username@192.168.10.30

Replace your_username with your DSM admin username and 192.168.10.30 with your NAS IP. Enter your password when prompted.

4

Create the deployment folder

Create a folder in /volume1/docker/ which is the standard location for Docker projects on Synology:

mkdir -p /volume1/docker
💡
Tip: Find your NAS IP address

In DSM, go to Control Panel → Network → Network Interface. Your IP will be listed next to LAN 1. It usually looks like 192.168.x.x.

6. Cloning the Repository

Downloading the application code onto the NAS.

The application code lives on GitHub. "Cloning" means downloading a full copy of it to the NAS. Run these commands over SSH: 

cd /volume1/docker
sudo git clone https://github.com/MCCI-git/printer-asset-management.git
sudo chown -R your_username:users /volume1/docker/printer-asset-management
cd /volume1/docker/printer-asset-management

Line by line explanation:

CommandWhat it does
cd /volume1/dockerNavigate into the docker folder
sudo git clone ...Download the entire repository. sudo is needed because /volume1/docker requires elevated permissions.
sudo chown -R ...Transfer ownership of the downloaded files to your user. Without this you can't edit the files or access them in File Station.
cd printer-asset-managementEnter the project folder — all following commands run from here.
⚠️
Always fix ownership after cloning with sudo

When you use sudo git clone, the files are created owned by root. The chown command transfers them to your user so you can access and edit them normally. Skipping this step causes "Permission denied" errors.

💡
Tip: Combine both commands

You can chain them so ownership is fixed immediately after cloning:

sudo git clone https://github.com/MCCI-git/printer-asset-management.git \
  /volume1/docker/printer-asset-management && \
  sudo chown -R your_username:users /volume1/docker/printer-asset-management

7. Environment Configuration

The .env file — secrets and settings that are never committed to GitHub.

The .env file contains sensitive configuration: database passwords, the app URL, and encryption keys. It is never uploaded to GitHub — each deployment creates its own. A template called .env.prod.example is provided as a starting point.

1

Copy the template

cp .env.prod.example .env
2

Generate a Laravel APP_KEY

This is a unique encryption key for your installation. Generate one with:

echo "base64:$(openssl rand -base64 32)"

Copy the output — it will look like: base64:3fiInZz22meE...hXg=

3

Edit the .env file

vi .env

Fill it in with your values:

APP_KEY=base64:YOUR_GENERATED_KEY_HERE
APP_URL=http://192.168.10.30:8900
APP_PORT=8900

DB_CONNECTION=mysql
DB_HOST=pam_db
DB_PORT=3306
DB_DATABASE=printer_assets
DB_USERNAME=pam_user
DB_PASSWORD=$$YourStrongPassword2024
DB_ROOT_PASSWORD=$$YourRootPassword2024
🔴
Critical: Escape the $ sign in passwords

Docker Compose reads the .env file and treats ${...} as variable substitution. If your password contains a $ character (e.g. $ecret), you must write it as $$ecret (double dollar sign). Otherwise Docker will silently mangle your password and the database won't connect.

What each variable means

VariableExample valueDescription
APP_KEYbase64:abc123...Laravel's encryption key. Must be unique per installation. Never share this.
APP_URLhttp://192.168.10.30:8900The full URL where the app is accessible. Use your NAS IP.
APP_PORT8900The port Nginx listens on. Change if 8900 is already in use.
DB_CONNECTIONmysqlDatabase driver. Must be mysql — do not change.
DB_HOSTpam_dbThe database container name. Docker resolves this automatically.
DB_PASSWORD$$MyPasswordPassword for the pam_user database user. Remember to escape $.
DB_ROOT_PASSWORD$$RootPasswordMySQL root password. Used only during first setup.
💡
Tip: vi editor basics

When editing files with vi: press i to start typing, use arrow keys to move, press Esc when done typing, then type :wq and press Enter to save and exit. To exit without saving: :q!

8. Docker Compose Configuration

Understanding how the containers are defined and connected.

Before starting the containers, create a docker-compose.yml from the production template. This is the file Docker Compose actually reads: 

cp docker-compose.prod.yml docker-compose.yml

The file defines 4 services. Here is each one explained:

Service 1: pam_db (MySQL Database)

pam_db:
  image: mysql:8.0           # Uses the official MySQL 8 image from Docker Hub
  container_name: pam_db     # Fixed name so other containers can refer to it
  environment:
    MYSQL_DATABASE: printer_assets   # Creates this database on first run
    MYSQL_USER: pam_user             # Creates this user
    MYSQL_PASSWORD: ${DB_PASSWORD}   # Password comes from .env
    MYSQL_ROOT_PASSWORD: ${DB_ROOT_PASSWORD}
  healthcheck:
    test: ["CMD", "mysqladmin", "ping"]
    retries: 30
    start_period: 300s       # Wait up to 5 min for MySQL to initialise on first run
ℹ️
Why start_period: 300s?

On first run, MySQL initialises its data directory which can take 3–5 minutes on NAS hardware. The start_period tells Docker to not count healthcheck failures during this initialisation window. Without it, Docker marks the container as unhealthy and aborts startup.

Service 2: pam_backend (Laravel / PHP)

pam_backend:
  build:
    context: ./backend
    dockerfile: Dockerfile.prod   # Builds from backend/Dockerfile.prod
  environment:
    DB_CONNECTION: mysql
    DB_HOST: pam_db        # Docker resolves "pam_db" to the database container IP
    APP_KEY: ${APP_KEY}    # Passed from .env
  depends_on:
    pam_db:
      condition: service_healthy  # Only starts after database is ready

Service 3: pam_nginx (Nginx + React Frontend)

pam_nginx:
  build:
    context: .
    dockerfile: Dockerfile.nginx  # Multi-stage: builds React, then serves with Nginx
  ports:
    - "${APP_PORT:-8900}:80"       # Maps NAS port 8900 to container port 80
  depends_on:
    - pam_backend

Service 4: pam_phpmyadmin (Database browser — optional)

pam_phpmyadmin:
  image: phpmyadmin:latest
  ports:
    - "8901:80"            # Access at http://NAS-IP:8901
  environment:
    PMA_HOST: pam_db       # Connects to the database container

How Nginx routes requests

The nginx.prod.conf file tells Nginx what to do with each request:

URL patternAction
/api/*Forward to Laravel via FastCGI (PHP-FPM)
/sanctum/*Forward to Laravel (authentication handshake)
/storage/*Forward to Laravel (uploaded files)
Everything elseServe the React index.html (SPA routing)

9. Building and Starting the Containers

Going live for the first time.

ℹ️
Make sure you are in the project folder

All commands must be run from inside /volume1/docker/printer-asset-management. If you get "no configuration file provided", you are in the wrong directory.

1

Build all container images

This compiles the PHP backend, builds the React frontend, and assembles the Nginx image. It downloads dependencies from the internet so it takes a few minutes.

sudo docker compose build --no-cache

--no-cache forces a fresh build, ignoring any previously cached layers. Always use this flag after pulling updates.

2

Start all containers

sudo docker compose up -d

-d means "detached" — containers run in the background. Without it they would stop when you close the terminal.

3

Monitor startup

Check the status of all containers:

sudo docker compose ps

You are waiting for this output (MySQL takes up to 5 minutes on first run):

pam_db          Up (healthy)
pam_backend     Up
pam_nginx       Up    0.0.0.0:8900->80/tcp
⚠️
Do not proceed until pam_db shows "(healthy)"

If you run migrations before MySQL is ready, they will fail. Wait for the healthy status.

4

Verify the build worked

Open a browser and go to http://192.168.10.30:8900. You should see the login page. If you see a blank page or error, check the logs:

sudo docker logs pam_nginx --tail 30
sudo docker logs pam_backend --tail 30

10. Database Setup

Creating tables and your first admin account.

1

Run database migrations

Migrations create all the database tables. This must be run once on a fresh installation and again after any update that includes schema changes.

sudo docker exec -it pam_backend php artisan migrate --force

You should see a list of migrations running with DONE next to each one.

2

Create your first admin user

Open the Laravel Tinker interactive console:

sudo docker exec -it pam_backend php artisan tinker

Then paste this command on a single line (replace the values):

\App\Models\User::create(['name' => 'Your Name', 'email' => 'you@example.com', 'password' => bcrypt('yourpassword'), 'role' => 'super-admin', 'status' => 'active']);

Press Ctrl+D to exit Tinker.

⚠️
Passwords with $ in Tinker

If your password contains a $ character, you must escape it as \$ inside double quotes when typing in Tinker. Example: bcrypt("\$ecret123"). The easiest workaround is to use a password without $ for initial setup, then change it from the app's profile settings once logged in.

3

Log in

Open http://192.168.10.30:8900 in your browser and log in with the email and password you just created.

12. Application Features

A walkthrough of each module.

Printers

The Printers page is the core of the app. Each printer record stores its model, serial number, location, department, status, IP address, last and next service dates, and the user it is assigned to.

  • Add a printer: Click the Add Printer button. Fill in the form and save.
  • Edit: Click the pencil icon on any row.
  • Show details: Click the eye icon to open a full detail panel showing page count history and assigned consumables.
  • Page counts: Inside the details panel, log monthly page counts to track usage over time.
  • Status: Active, Under Maintenance, or Retired.

Consumables

Track toner cartridges, drums, fuser units, and any other consumable items.

  • Stock levels: Set minimum stock thresholds. The dashboard alerts you when stock falls below the minimum.
  • Assigning: Use the assign button to link a consumable to a specific printer. The assignment date is recorded automatically.
  • Unassigning: Remove the assignment when the item is replaced. The consumable disappears from the printer's detail panel in real time.

Maintenance (Work Orders)

Each maintenance job is a Work Order with a unique reference number (e.g. WO-001). Work Orders have a status: Open, In Progress, or Completed. When a Work Order is marked Completed, the completion date is recorded and shown in the table.

Activity Log (Reports)

Every change made in the app — creating a printer, updating a contract, assigning a consumable — is automatically recorded in the Activity Log. This provides a full audit trail. You can filter by date, model type, and action type, and export to CSV.

Budget

Set a total annual budget split into CAPEX (capital purchases) and OPEX (running costs). The app automatically calculates actual spend from your recorded purchases and shows the remaining budget in real time.

Email Alerts (Admin)

Configure an SMTP server (e.g. your company mail server or Gmail SMTP) in Admin → SMTP Settings. Then set up notification thresholds in Admin → Notifications:

  • Low stock alerts
  • Out of stock alerts
  • Contract expiry warnings (X days before)
  • Overdue service alerts

14. Updating the Application

How to deploy new versions of the app.

Updates are pushed to GitHub by the developer. To apply them to your NAS, you pull the changes and rebuild the containers. Your data is safe — it lives in a Docker volume that is never touched during an update.

Update procedure

1

SSH into the NAS

ssh your_username@192.168.10.30
2

Navigate to the project folder

cd /volume1/docker/printer-asset-management
3

Pull the latest code from GitHub

git pull
4

Rebuild and restart

sudo docker compose build --no-cache pam_backend pam_nginx
sudo docker compose up -d

Only rebuild the services that changed. If only the backend changed, you can omit pam_nginx to save time.

5

Run any new migrations

sudo docker exec -it pam_backend php artisan migrate --force

This is safe to run even if there are no new migrations — it will simply report that everything is up to date.

💡
Your data is never deleted during updates

The database lives in a Docker volume called pam_mysql_data. This volume persists across container rebuilds and docker compose down commands. Your data is only deleted if you explicitly run docker volume rm pam_mysql_data — which you should never do in production.

16. Troubleshooting

Common issues and how to fix them.

🔴 "Permission denied" when cloning or accessing files

Cause: Files were created by root when using sudo git clone.

Fix:

sudo chown -R your_username:users /volume1/docker/printer-asset-management

🔴 MySQL container stays "unhealthy" for a long time

Cause: First-run database initialisation takes 3–5 minutes on NAS hardware.

Fix: Wait. Check the healthcheck config has start_period: 300s and retries: 30 in docker-compose.yml. Do not restart the container — just wait.

🔴 "Database file at path does not exist" (SQLite error)

Cause: DB_CONNECTION=mysql is missing from the container's environment.

Fix: Ensure DB_CONNECTION: mysql is in the pam_backend environment block of docker-compose.yml and that DB_CONNECTION=mysql is in your .env file. Then restart:

sudo docker compose restart pam_backend

🔴 npm ci fails during build (lock file mismatch)

Cause: The package-lock.json was generated on a different platform and is out of sync.

Fix: Regenerate it using Docker so it matches the build environment:

cd /volume1/docker/printer-asset-management/frontend
sudo docker run --rm -v "$(pwd)":/app -w /app node:20-alpine npm install --package-lock-only
cd ..
sudo docker compose build --no-cache pam_nginx

🔴 PHP version error during backend build

Cause: backend/Dockerfile.prod uses PHP 8.3 but packages require PHP ≥ 8.4.

Fix: The Dockerfile.prod should already have FROM php:8.4-fpm-alpine. If not, update it:

sed -i 's/php:8.3-fpm-alpine/php:8.4-fpm-alpine/' backend/Dockerfile.prod

🔴 Login shows "CSRF token mismatch"

Cause: Laravel's stateful middleware was enabled, conflicting with token-based auth.

Fix: Ensure backend/bootstrap/app.php does not contain $middleware->statefulApi();. Rebuild after removing it.

🔴 "no configuration file provided: not found"

Cause: You are not in the project directory.

Fix:

cd /volume1/docker/printer-asset-management

🔴 App loads but API calls fail (404 on /api/)

Cause: Nginx config is incorrect or stale.

Fix: Rebuild the nginx container:

sudo docker compose build --no-cache pam_nginx && sudo docker compose up -d

🔴 Containers stopped after NAS reboot

Cause: The restart: unless-stopped policy is set, so containers should auto-start. If they don't:

Fix:

cd /volume1/docker/printer-asset-management && sudo docker compose up -d

Getting more information

When something goes wrong, these two commands give the most useful information:

# See what Laravel logged
sudo docker exec pam_backend tail -100 /var/www/html/storage/logs/laravel.log

# See what the container printed to its console
sudo docker logs pam_backend --tail 50
💡
Tip: Always check the logs first

Before trying any fix, read the error in the logs. The error message almost always tells you exactly what is wrong. Guessing without reading the logs wastes time.
