Overview & Architecture

The application is made up of three containers that work together:

ContainerWhat It DoesTechnology
pam_nginxThe front door — serves the web pages and passes API calls to the backendNginx + React
pam_backendThe brain — handles all business logic, user authentication, and database queriesPHP / Laravel
pam_dbThe memory — stores all your printers, contracts, consumables, suppliers dataMySQL 8

All three containers are on a private internal network (pam_net). Only Nginx is exposed to your local network on one port.

Simple Analogy Think of it like a restaurant: Nginx is the front of house (takes your order), Backend is the kitchen (prepares it), and the Database is the fridge (stores the ingredients).

Prerequisites

  • Synology NAS with DSM 7.2 or later
  • Container Manager installed from Package Center
  • SSH access to the NAS enabled (Control Panel → Terminal & SNMP → Enable SSH)
  • A computer on the same network as the NAS
  • The project folder: Printer Asset Management/

Step 1 — Verify Port Numbers

Before deploying, you must make sure the port you pick isn't already used by another container or service on your NAS. The app is pre-configured to use port 8900.

How to check used ports

1
Open Container Manager Go to Container Manager → Container. Look through the list of running containers and note their port mappings. Ports are shown in the format HOST:CONTAINER.
2
Check via SSH (more reliable) SSH into your NAS and run: 
# Lists all ports currently in use by Docker containers
sudo docker ps --format "table {{.Names}}\t{{.Ports}}"
3
If port 8900 is taken Edit the .env.prod file and change APP_PORT=8900 to any free port (e.g. 8901, 9000, 9200). Avoid ports below 1024 and common ones like 80, 443, 5000, 8080.
Common Ports to Avoid 80/443 (web), 5000 (DSM), 5001 (DSM HTTPS), 8080 (often used), 3306 (MySQL). Your PAM database runs on MySQL internally but is NOT exposed to your network — so 3306 is safe internally.

Step 2 — Prepare the Project Files

Do these steps on your computer (not the NAS).

1
Create the environment file In the root of the project folder, copy .env.prod.example and rename it to .env.prod. Open it and fill in: 
APP_KEY=base64:GENERATE_BELOW
APP_URL=http://YOUR_NAS_IP:8900
APP_PORT=8900
DB_PASSWORD=choose_something_strong_123
DB_ROOT_PASSWORD=another_strong_password_456
2
Generate the APP_KEY This is a security key for encrypting data. Run this on your computer (requires PHP): 
php -r "echo 'base64:' . base64_encode(random_bytes(32)) . PHP_EOL;"
Copy the output and paste it as the value for APP_KEY in your .env.prod file.

If you don't have PHP installed locally, you can generate it on the NAS after deployment — instructions in Step 5.
3
Replace your NAS IP Find your NAS IP address (usually something like 192.168.1.100). Go to DSM → Control Panel → Network to find it. Replace YOUR_NAS_IP in APP_URL with your actual IP.

Step 3 — Transfer Files to NAS

1
Create a folder on your NAS Open File Station on the NAS. Inside the docker shared folder, create a new folder called printer-asset-management.
2
Copy the project to the NAS Use File Station's upload button, or use the following command from your computer's terminal (replace the IP): 
scp -r "Printer Asset Management/" admin@192.168.1.100:/volume1/docker/printer-asset-management/
It will ask for your NAS admin password.
3
What to exclude (save space) You can skip these folders — they are not needed on the NAS:
  • frontend/node_modules/ (very large, not needed — Docker rebuilds it)
  • backend/vendor/ (same reason)

Step 4 — Deploy with Container Manager

Two Ways to Deploy Option A (Recommended) is via SSH using the command line — faster and more reliable. Option B is through the Container Manager GUI.

Option A — SSH (Recommended)

1
Connect to your NAS via SSH On your computer, open Terminal (Mac/Linux) or PowerShell (Windows) and type: 
ssh admin@192.168.1.100
Enter your NAS admin password when prompted.
2
Navigate to the project folder 
cd /volume1/docker/printer-asset-management/"Printer Asset Management"
3
Build and start the containers 
# Build all images (this takes 3-5 minutes the first time)
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod build

# Start all containers in the background
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod up -d

Option B — Container Manager GUI

1
Open Container Manager → Project → Create
2
Set the project path to /docker/printer-asset-management/Printer Asset Management
3
Select the compose file: docker-compose.prod.yml
4
Set environment file: .env.prod
5
Click Build and then Deploy. Monitor the build log — it will take a few minutes.

Step 5 — First-Run Setup

After the containers are running, you need to initialise the database once.

1
Wait 30 seconds for MySQL to fully start up, then run the database migrations: 
sudo docker exec pam_backend php artisan migrate --force
You should see a list of migrations completing successfully.
2
Create your first admin user 
sudo docker exec -it pam_backend php artisan tinker

# Inside tinker, paste this (replace values):
App\Models\User::create([
  'name'     => 'Your Name',
  'email'    => 'your@email.com',
  'password' => bcrypt('YourPassword'),
  'role'     => 'super-admin',
  'status'   => 'active',
]);
# Press Enter, then type: exit
3
Clear and cache the config 
sudo docker exec pam_backend php artisan config:cache
sudo docker exec pam_backend php artisan route:cache
sudo docker exec pam_backend php artisan view:cache
This makes the app faster by caching configuration.
4
Link the storage folder (needed for uploaded logos and PDFs) 
sudo docker exec pam_backend php artisan storage:link

Step 6 — Verify the Deployment

1
Check all containers are running 
sudo docker ps | grep pam_
You should see pam_nginx, pam_backend, and pam_db all showing status Up.
2
Open the app in your browser Go to: http://YOUR_NAS_IP:8900 You should see the Printer Asset Management login page.
3
Log in with the admin account you created in Step 5.
You're Live! If you can log in and see the dashboard, the deployment was successful. You can now access the app from any device on your network.

Step 7 — Configure Backups

There are two things to back up: the database (your data) and the storage files (uploaded logos, contract PDFs, SMTP config).

Database Backup

Run this command to create a database dump. Schedule it via Synology Task Scheduler.

# Creates a timestamped SQL backup file
sudo docker exec pam_db mysqldump -u pam_user -p'YOUR_DB_PASSWORD' printer_assets \
  > /volume1/docker/backups/pam_db_$(date +%Y%m%d_%H%M).sql

Schedule Automatic Daily Backups

1
Go to DSM → Control Panel → Task Scheduler → Create → Scheduled Task → User-defined script
2
Task name: PAM Database Backup
User: root
Schedule: Daily at 02:00
3
Script tab — paste this: 
#!/bin/bash
BACKUP_DIR="/volume1/docker/backups/pam"
mkdir -p $BACKUP_DIR
docker exec pam_db mysqldump -u pam_user -p'YOUR_DB_PASSWORD' printer_assets \
  > $BACKUP_DIR/pam_db_$(date +%Y%m%d_%H%M).sql

# Keep only the last 30 backups
ls -t $BACKUP_DIR/pam_db_*.sql | tail -n +31 | xargs -r rm

Storage Files Backup

The pam_storage Docker volume contains uploaded files. Back it up with:

sudo docker run --rm \
  -v pam_storage:/data \
  -v /volume1/docker/backups/pam:/backup \
  alpine tar czf /backup/pam_storage_$(date +%Y%m%d).tar.gz -C /data .
Off-Site Backup Recommendation Use Synology's Hyper Backup package to copy your /volume1/docker/backups/pam/ folder to an external USB drive or cloud storage (e.g., Backblaze, Google Drive) for disaster recovery.

Making Edits After Deployment

When you make code changes on your computer, follow these steps to push the update to the NAS.

For frontend changes (UI, pages, styling)

1
Make your changes on your computer as usual.
2
Copy the updated files to the NAS 
scp -r "Printer Asset Management/frontend/src/" \
  admin@192.168.1.100:/volume1/docker/printer-asset-management/"Printer Asset Management"/frontend/src/
3
Rebuild the Nginx container (this rebuilds the frontend): 
cd /volume1/docker/printer-asset-management/"Printer Asset Management"
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod build pam_nginx
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod up -d pam_nginx

For backend changes (API, controllers, models)

1
Copy updated backend files 
scp -r "Printer Asset Management/backend/app/" \
  admin@192.168.1.100:/volume1/docker/printer-asset-management/"Printer Asset Management"/backend/app/
2
Clear Laravel cache (no rebuild needed for PHP changes): 
sudo docker exec pam_backend php artisan config:clear
sudo docker exec pam_backend php artisan cache:clear

For database changes (new migrations)

# Copy the new migration file, then run:
sudo docker exec pam_backend php artisan migrate --force
Tip — Use a Script Save your most common update commands in a shell script called deploy-update.sh so you can run the whole update with one command.

Monthly Checks (Manual)

  • Check logs for errors: sudo docker logs pam_backend --tail 100
  • Check disk usage: Container Manager → Volume — ensure the pam_mysql_data volume isn't growing unexpectedly
  • Update Docker images (optional): Pull new base images for security patches
  • Review user accounts in the Admin panel — disable accounts for staff who have left
  • Verify backups — restore a test backup to confirm integrity

Updating the App (Future Versions)

1
Take a database backup first (always before updating)
2
Copy new files to the NAS (overwrite existing)
3
Rebuild and restart containers: 
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod build
sudo docker compose -f docker-compose.prod.yml --env-file .env.prod up -d
4
Run any new migrations: 
sudo docker exec pam_backend php artisan migrate --force

Troubleshooting

ProblemLikely CauseFix
Can't open the app in browser Port conflict or container not running Run sudo docker ps | grep pam_nginx — check it shows "Up". Check the port in .env.prod.
Login fails with 500 error Missing APP_KEY or database not connected Check sudo docker logs pam_backend --tail 50 for the error message.
Blank white screen Frontend build error Check sudo docker logs pam_nginx. Rebuild with docker compose build pam_nginx.
Database container keeps restarting Wrong password in .env.prod Check sudo docker logs pam_db. Correct the password and re-run docker compose up -d.
Uploaded logos don't appear Storage link not created Run sudo docker exec pam_backend php artisan storage:link
SMTP email not working Gmail App Password issue Go to Admin panel, re-enter your 16-character App Password, save, and test.
Never Do This Do not run docker compose down -v — the -v flag deletes all volumes including your database. Use docker compose down (without -v) to safely stop containers.
